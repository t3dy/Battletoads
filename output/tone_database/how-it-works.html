How It Works

From ROM bytes to REAPER projects — the complete NES music extraction pipeline.

The NES Sound Chip (RP2A03)

The NES has no sound samples, no wavetables, no DSP. All audio is generated by 5 simple waveform generators running at 1.789773 MHz, updated 60 times per second by the CPU. Every sound you've ever heard in an NES game — every melody, every bass line, every drum hit — is built from these 5 channels:

Pulse 1 & 2 (Square Waves)

NES HardwareModern Equivalent

Two identical pulse wave generators. Each has:

  • 11-bit period register → frequencies from ~55Hz to ~12.4kHz
  • 4 duty cycles: 12.5%, 25%, 50%, 75% (waveform shape)
  • 4-bit volume (0-15) with optional hardware decay
  • Sweep unit for automatic pitch bends

The CPU writes new values to these registers every frame (60Hz). That's how envelopes and vibrato work — not through hardware automation, but through the music driver manually updating registers 60 times per second.

In ReapNES Console (JSFX plugin for REAPER):

  • CC11 = Volume (mapped to NES 0-15 range)
  • CC12 = Duty cycle (0-3 selecting waveform shape)
  • MIDI note number = Pitch (quantized to semitones)
  • Slider 33 = Channel mode (0=Sq1, 1=Sq2)

The per-frame CC11/CC12 automation IS the NES envelope. Don't override it with ADSR — the captured data already contains the exact volume and timbre curve the game uses.

Triangle (Bass)

NES HardwareModern Equivalent

A 32-step staircase triangle wave. Unlike pulse channels:

  • No volume control — only on/off gating
  • One octave lower than pulse for the same period (32-step vs 16-step sequencer)
  • All expression comes from note duration (staccato vs legato)

The triangle wave contains only odd harmonics falling off as 1/n², making it the darkest, smoothest waveform the NES can produce. It's always used for bass.

In ReapNES Console:

  • Channel Mode = 2 (Triangle)
  • CC11 always 127 (full volume gate)
  • No CC12 (no duty cycle for triangle)
  • Articulation from MIDI note duration only

Noise (Percussion)

NES HardwareModern Equivalent

A linear-feedback shift register (LFSR) generating pseudo-random noise:

  • 16 period settings control noise 'brightness'
  • 2 modes: long (white noise) / short (metallic tonal)
  • 4-bit volume with hardware envelope
  • Every NES drum is SYNTHESIZED from noise — no samples

In ReapNES Console:

  • Channel Mode = 3 (Noise)
  • MIDI notes map to drums: 36=kick, 38=snare, 42=hi-hat
  • Velocity = initial volume
  • ADSR envelope shapes each hit

How We Extract the Music

NES ROM (.nes) → Headless Emulator (py65 6502 CPU) → APU Register Captures → Mesen CSV → MIDI → REAPER Project

Step 1: Boot the ROM

nes_rom_capture.py loads the ROM, emulates the 6502 CPU, handles mapper bank switching (MMC1, MMC3, UxROM, etc.), and simulates NMI interrupts at 60Hz. No graphics — we only care about writes to the APU registers at $4000-$4017.

Step 2: Capture APU Writes

Every time the CPU writes to an APU register, we record it: frame number, register address, value. This is exactly what Mesen (a real NES emulator) captures — our output is format-compatible.

Step 3: Convert to MIDI

mesen_to_midi.py translates the register captures into standard MIDI: period changes become note-on/note-off events, volume writes become CC11, duty cycle writes become CC12. The result is a MIDI file that encodes the exact per-frame state of every NES channel.

Step 4: Generate REAPER Project

generate_project.py creates a REAPER DAW project with 4 tracks (one per NES channel), each loaded with the ReapNES Console JSFX synthesizer configured for the correct channel mode. Open the .rpp file in REAPER, press play, hear the NES music.

The Three-Priority Cascade

The ReapNES synthesizer has three input modes, auto-selected by what data arrives:

Priority 1: SysEx Register Replay (Maximum Fidelity)

Raw APU register state encoded in MIDI SysEx messages. The synth becomes a software NES APU — it reads the registers and generates the exact waveform at the sample rate. Every hardware behavior is reproduced: sweep, phase reset, noise LFSR mode. This is the NES hardware running in software.

Priority 2: CC11/CC12 (File Playback)

Volume from CC11 (0-127 → NES 0-15), duty from CC12 (0-3), pitch from MIDI notes. This is what our ROM-extracted files use. Loses sweep unit detail and sub-semitone pitch, but captures the essential envelope and timbre perfectly.

Priority 3: ADSR Keyboard (Live Composing)

When no file data arrives, the synth uses its own ADSR envelope to shape notes from a MIDI keyboard. Per-game presets capture each game's characteristic attack/decay/sustain/release values. This is for composers who want to write new music in the style of a specific NES game.
