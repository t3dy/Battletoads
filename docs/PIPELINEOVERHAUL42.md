# Pipeline Overhaul: What We Learned and How It Should Work

**Date**: 2026-04-02
**Context**: Battletoads rip sessions v1-v6, CV1/Contra methodology review

---

## The Problem We Were Solving Wrong

We had the right data (Mesen trace with 9,495 frames of APU register state)
and the right synth (Console with CC11/CC12 playback). But we kept trying
to skip the interpretation step between them:

**What we did (wrong):**
```
Mesen trace → raw period changes = MIDI notes → synth
```
Every time the APU period register changed, we created a new MIDI note.
Result: sweep vibrato became trills, attack arpeggios became rapid-fire
notes, and the output sounded nothing like the game.

**What CV1 and Contra did (right):**
```
ROM commands → parser → Frame IR → MIDI notes + CC envelope → synth
```
The parser understood the driver's intent. The Frame IR shaped the
envelopes. The MIDI only contained real musical events.

**What we need for Battletoads:**
```
Mesen trace → Frame IR interpretation → stable notes + CC envelope → synth
```
Since we don't have ROM parsing for Battletoads (different driver, not
Konami Maezawa), the Frame IR must be built from the trace data by
detecting what the hardware is doing and interpreting it musically.

## The Proven Pipeline (from CV1/Contra)

### Layer 1: Source Data
- **CV1/Contra**: ROM parsing (driver commands decoded)
- **Battletoads**: Mesen trace capture (per-frame APU register state)
- **NSF**: 6502 emulation (unreliable for Battletoads — broken triangle,
  broken noise, impossible notes below NES minimum)

### Layer 2: Frame IR (Interpretation)
This is the critical step we were skipping. It converts raw register
state into musical meaning:

| Raw Register Event | Musical Interpretation |
|---|---|
| Period changes every frame for 8 frames spanning 3 octaves | Attack transient (single percussive "thwack") |
| Period oscillates ±4 around a center for 20+ frames | Sustained note with hardware sweep vibrato |
| Period jumps to new value and holds for 5+ frames | Real note change |
| Volume decays 15→14→13→...→0 over 10 frames | Envelope decay (CC11 automation) |
| Linear counter > 0 AND period > 2 on triangle | Triangle gate ON |
| Noise volume rises from 0 while period changes | Drum hit |

**The rule**: A real note must hold the same MIDI note number for at
least 3 consecutive frames (50ms). Anything shorter is an artifact of
hardware behavior (sweep unit, rapid arpeggios, SFX) and gets collapsed
into the nearest stable note.

### Layer 3: MIDI Encoding
After interpretation, the MIDI contains:
- **Note events**: Only from stabilized pitches (no sweep trills)
- **CC11**: Volume per-frame (ground truth envelope from APU)
- **CC12**: Duty cycle per-frame (timbre changes)
- **Noise**: Drum hits from volume rising-edge + period-change detection

### Layer 4: REAPER Project
- **Console synth** (ReapNES_Console.jsfx) reads CC11/CC12 for playback
- ADSR mode available for keyboard composing
- Full RPP header, REC 5088, HASDATA embedding
- Generated by `generate_project.py` only

### Layer 5: Human Ear-Check
- Compare to game audio (not NSF — NSF is unreliable for Battletoads)
- Fix order: pitch → timing → volume → timbre
- One hypothesis per test cycle

## What Each Script Does

### `trace_to_midi.py` (the main pipeline for trace-based games)
```
Input:  Mesen capture CSV
Output: MIDI (with CC11/CC12) + REAPER projects (Console synth)
```
1. `parse_mesen_csv()` — raw per-frame channel data
2. `filter_sfx()` — remove ultrasonic artifacts (conservative)
3. `build_trace_midi()` — **smart note detection** (Frame IR step):
   - Computes MIDI note per frame from period
   - Finds stable notes (same MIDI note for 3+ frames)
   - Collapses sweep/arpeggio artifacts into target notes
   - Emits CC11/CC12 per-frame for ground-truth envelopes
   - Detects noise hits from volume edges + period changes
4. `process_segment()` — saves MIDI, calls `generate_project.py`

### `nsf_to_reaper.py` (for NSF-based games where NSF is reliable)
```
Input:  NSF file
Output: MIDI + REAPER projects + WAV previews
```
Uses 6502 CPU emulation to capture APU state, then `build_midi()` to
create MIDI with per-frame CC11/CC12. Works well for games where NSF
matches the actual game audio (most games). Not reliable for Battletoads
or Mario (proven divergence).

### `generate_project.py` (REAPER project builder)
```
Input:  MIDI file + synth choice
Output: REAPER .RPP with embedded MIDI and synth configuration
```
Never write RPP by hand. This script handles all the REAPER configuration
that took weeks to get right (HASDATA, REC 5088, channel modes, etc.).

## Route Decision: When to Use Which Pipeline

| Condition | Route | Script |
|---|---|---|
| NSF exists AND no trace | NSF | `nsf_to_reaper.py` |
| Trace exists AND no NSF | Trace | `trace_to_midi.py` |
| Both exist, NSF matches game | NSF (faster) | `nsf_to_reaper.py` |
| Both exist, NSF diverges | Trace | `trace_to_midi.py` |
| Unknown | Try NSF first, compare to game | Both, A/B test |

**Confirmed trace-required games**: Battletoads, Super Mario Bros.
**Confirmed NSF-reliable games**: (TBD — need systematic testing)

## The SysEx Question (Resolved)

We built a SysEx register replay path (APU2 synth) that drives oscillators
directly from raw register bytes. In theory, this bypasses all MIDI
encoding limitations. In practice:

1. The SysEx path reproduces hardware behavior too faithfully — sweep
   artifacts, 1-frame arpeggio attacks, and ultrasonic periods all
   produce audible artifacts that NES analog output smooths over.
2. The Console synth + CC11/CC12 path produces better results because
   the Frame IR interpretation step filters these artifacts.
3. SysEx may be useful later for the unified synth (ReapNES Studio)
   which could include analog-style filtering. For now, CC is the path.

**Decision**: Use Console synth + CC11/CC12 for all production output.
Keep SysEx generation in the pipeline for future use, but the primary
deliverable is always the Console RPP.

## Battletoads-Specific Findings

1. **NSF is broken**: Triangle has 22 notes (should be 414), noise has
   1 note (should be 223), pulse has impossible notes below A1.
2. **Trace is ground truth**: 9,495 frames (158.2s) of Level 1 gameplay
   with full APU register state.
3. **The sweep "thwack"**: P1 does rapid 8-frame descending sweeps
   (D#4→C#2) as attack transients. These must be collapsed into single
   notes, not played as trills.
4. **P2 vibrato**: Period oscillates ±4 around center (~677 = E3).
   All values map to the same MIDI note. Not a problem in CC path.
5. **Triangle period**: Doesn't get set until frame 524. Earlier linear
   counter activity (frames 3-24, 255-264) has period=0 = correctly silent.
6. **The capture has multiple loops**: 158s at ~67s per loop = ~2.3 loops.
   Clean extraction should use one loop cycle.
